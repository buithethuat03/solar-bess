## 1. Quality objectives

Prove business behavior, tenant/permission integrity, exact calculation, immutable/workflow state, file safety, integration idempotency/reconciliation, recovery and no OT control. Testing is risk-based but no critical security/safety/financial invariant can be waived merely to meet a percentage.

## 2. Test levels and responsibility

| Level | Focus | Owner/evidence |
|---|---|---|
| Unit/property | Domain invariant, calculations, policy predicates, parsers | Engineering; deterministic automated results |
| Component | Module command/state/error/outbox | Module team; isolated dependencies |
| Integration | DB/object/search/queue/connector/time-series | Engineering/QA; contract/reconciliation |
| API/contract | OpenAPI schema, status/error/idempotency/auth | QA/API owner; automated |
| UI/component | States/accessibility/localization/responsive | UX/Frontend/QA |
| End-to-end | Cross-domain user journey/workflow | QA/Process Owner |
| Permission/tenant | RBAC/ABAC/SoD and cross-scope negative | Security/QA; zero leakage |
| Security | Threat controls, DAST/fuzz/file/supply chain | Security/Engineering |
| Performance/load | NFR workload/latency/fairness | Performance QA/SRE |
| Resilience/recovery | Retry/DLQ/failure/backup/DR | SRE/QA/System Owners |
| OT integration | Gateway/tag/time/quality/gap/replay/no-control | OT Owner/Security/QA |
| UAT | Business outcome, evidence and usability | Product/Process users |

## 3. Test environments

| Environment | Purpose | Data/integration |
|---|---|---|
| Local/unit | Fast deterministic tests | Synthetic/fakes; no real secret/PII |
| Development | Component/API exploratory | Ephemeral tenant, stubs |
| Test/QA | Functional/regression/permission | Multi-tenant synthetic dataset; connector simulators |
| EC2 test — operational foundation | PostgreSQL/Redis/BullMQ/worker failure injection, migration và runtime smoke | Synthetic/resettable tenant data đã được Product Owner cho phép; không phải production/UAT evidence |
| Integration | Provider contract/reconciliation | Approved sandbox, masked synthetic payload |
| Performance | Isolated reproducible load | Generated volume/profile; production-like topology ratio |
| Staging/UAT | End-to-end/UAT/recovery rehearsal | Masked/synthetic; approved external sandbox |
| OT lab/digital simulator | Telemetry/alarm/gateway/network negative | No production controller command; one-way emulation |
| DR exercise | Restore/failover/failback | Isolated copies, recovery manifest |

No production data is copied unless approved masking/legal/security process exists. OT lab has no route to production controllers.

## 4. Test data design

Create at least Tenant A/B; multiple Company/LegalEntity; Project/Site/Package; internal/external roles; PM requester and independent approver; delegation active/expired; signed/issued/held documents; contracts/appendices/payment components in VND/USD; supplier/bid isolation; partial/damaged/duplicate serial; workfront/PTW/stop-work; inspection/NCR/punch; failed/retest; COD gate/waiver/handover; Solar/BESS asset hierarchy; fresh/stale/gap/replay telemetry; AI citations/confidence/rejection. US-003 fixtures include explicit Site timezone/calendar, schedules and alerts. US-004 adds more Risks than API-143 page limit across multiple scoring/threshold versions with complete/missing residual; fixed-role user.read matrix; Action for every API-149 branch plus mixed terminal payload and pre-command-owner SoD attack; matching/stale residualRiskVersion; Risk/Issue with two closure cycles after reopen/re-close; approved Change provenance; DB-105 rows/invalid candidates for every source→alert mapping and date/version derivation. Every dataset has owner, generation version, classification and reset.

## 5. Entry and exit criteria

### Entry

Approved requirement/version; environment/build/config identifiable; dependencies/stubs ready; test data and accounts/scopes; AC and expected state/error; defect baseline; security/OT approval for special tests.

### Exit

- 100% planned non-waivable tests executed and passed.
- No open Severity 1 or 2; Severity 3/4 accepted with owner/date.
- AC/NFR/SEC trace complete; regression and restore/security evidence attached.
- UAT proposed pass ≥95% of applicable scenarios, but all non-waivable scenarios 100%; exact threshold/waiver authority is Open Question.
- Performance/SLO targets that remain TBD cannot be marked passed.
- No cross-tenant leak, unsafe file release, self-approval, immutable overwrite or OT control path.

## 6. Defect severity

| Severity | Definition | Examples | Release |
|---|---|---|---|
| Sev 1 Critical | Safety/security/financial integrity or total critical outage | Tenant leak, OT command path, bypass stop-work, corrupt paid/signed data | Block |
| Sev 2 High | Major core flow unavailable/wrong without safe workaround | Wrong payment/Health/COD, lost event, malware release | Block |
| Sev 3 Medium | Material issue with safe workaround | Report/filter/workflow delay, isolated UX/accessibility | Owner decision |
| Sev 4 Low | Cosmetic/minor documentation/usability | Copy/layout/noncritical visual | May defer with owner |

## 7. Regression and automation

Automate unit/property, policy, API contract, state transitions, idempotency/concurrency, file pipeline, cross-tenant matrix, calculations and critical E2E. Use contract tests/provider simulators, deterministic clocks/currency/FX/timezone and seeded random/property tests. Visual tests do not replace accessibility/manual. OT/control negative search runs on OpenAPI, routes, schemas, UI/config and network evidence. Flaky tests are defects, quarantined only with owner/due.
